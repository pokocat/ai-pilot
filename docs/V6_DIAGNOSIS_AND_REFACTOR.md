## 一、V6.0 文档到底是什么（19 部分精读）

文档不是一份功能清单，而是一套**有状态的产品操作系统**。它的灵魂是一句话：

> **看清天势、市势、人势，在三势交汇处落子。**（§1 哲学内核「势」）

军师的终极形态被定义为老板的 **「外置大脑 + 灵魂镜子」**（§2）——它**比老板自己更记得、更懂他**。这意味着产品的核心价值不在"会聊天"，而在**持续累积的结构化状态**：命盘、决策日志、预言命中率、KPI 趋势、连续复盘天数、段位。把 19 部分按职能收敛成 6 层：

| 层 | 文档章节 | 本质 |
|---|---|---|
| **人格层** | §1 势 / §2 五层能力·角色 / §17 五种语气 / §18 语录库 | 「势」一元论统一四套思想；角色随老板**内在状态**（生存焦虑/增长兴奋/管理痛苦/瓶颈迷茫/意义追问）切换成 教官/参谋长/大哥/战略家/师父 |
| **天势/命理引擎层** | §4 八字全流程 | 采集八字→**四层推演**（命格真相/流年大运/性格手册/天时×战略融合）→命运叙事线→年度谶语→决策择时→命格×团队匹配→**预言验证**。命理结论翻译成比喻，禁用术语 |
| **诊断·规划层** | §3 智能路由 / §5 战略十二问 / §6 六轮+三轮 / §7 阶段自适应 / §8 行业知识库 / §14 思想武器 / §15 自检 / §16 防呆 | 入口识别→模式路由；十二问按核心矛盾动态排序；分阶段（生存/起步/成长/扩张）切换深度 |
| **陪跑·状态层（外置大脑的数据底座）** | §9 决策日志+战略档案 / §10 六层复盘 / §13 数据看板 | 决策日志#N（理由/天势参考/验证标准/验证周期/状态）；六层复盘（日/周/月/季/年/团队）+对齐率+连续复盘天数；三层看板（态势/KPI地图/五维健康度）+天势联动 |
| **增长·留存层** | §11 上瘾机制 / §12 仪式感 | 悬念钩子、里程碑解锁（7/30/90/180/365 天）、战友见证、**战略段位（新兵→元帅，按连续天数+决策准确率+预言命中率门控）**、裂变"送你一卦"、开局/授旗/年度复盘仪式 |
| **交付层** | §19 交付物总表 + 卡片骨架 | **A 级报告**（7 章+天势研判框+谶语封面）+ **12 种 B 级卡片**（每日/周/月/季战报、紧急决策、天命速写、晋升卡、性格操作手册、天时日历…），各有专属字段与版式 |

**一句话：V6.0 = 一个"永远在线、越用越懂你、用数据照镜子"的单一军师 + 一套靠"预言命中率/连续天数/段位"驱动的上瘾闭环。**

## 二、核心诊断：文档描述的是「有状态产品」，代码实现的是「无状态 prompt」

这是最重要的结论。证据如下（已逐项核对源码）：

### 2.1 身份错配——两套产品
- **文档**：一个持久的军师 = 外置大脑 + 灵魂镜子。
- **代码**：一个 **13 个 agent 的智库/工坊"市场"**（`server/src/data/agents.ts`：general/strat/growth/intel/fund/model/org/brand/ops + ip/promo/poster/shortvideo/copy），每个 agent **独立计费**（free/unlock/metered）。通用 chat + **一个通用 `ReportCard`** + 版本化报告。
- V6.0 只以**一段 41,713 字的 prompt** 存在于**生产 DB**（`agent.key=strat`，显示名「战略参谋[专业版]」，`pubVer=NULL` 无版本快照）。仓库里只有备份：`server/src/data/prompts/strat.v6.md`。
- ⚠️ **仓库里 seed 出来的 `strat` 是 3 行 businessPrompt stub**（`agents.ts:91-113`）。**按仓库代码跑（seed + mock 或 seed + 真模型），V6.0 连 prompt 都不存在**——它是"仅生产库里的一段文本"，比"prompt-only"还弱。

### 2.2 零状态底座
对 `schema.prisma`（40+ 模型）做关键词检索：**八字/命盘、决策日志、预言/命中率、连续复盘天数、段位、里程碑、命运叙事线、数据看板、复盘记录——0 张表、0 个字段、0 个计数器。** 唯一命中的是无关的 RAG `rerank` 和成本看板注释。

- 唯一的"持久化"是 `Profile.extraJson.dossier`——**一段手填、6000 字截断的自由文本 markdown**，被 `understanding.ts:172-183` 逐字塞进 prompt。
- `understanding.ts:16` 里 `RESERVED_EXTRA_KEYS = {dossier, bazi, 八字, decisionLog, prophecyLog}`——除 `dossier` 外**全是"占了名字但从不解析"的死缝**。
- **没有排盘/历法引擎**（`package.json` 无 lunar/ganzhi/紫微 依赖）；**没有调度器**（任何"久不复盘提醒/到期验证预言"都无法触发）；**没有写回闭环**（§9.2 承诺的"自动生成战略档案摘要并续接"实际上靠**让老板把那段文字复制粘贴回来**）。
- V6.0 prompt **0 占位符、无系统注入**，结构化上下文只能靠 `buildSystemParts` 无条件追加的【客户档案】块进来。

### 2.3 让 LLM 做它做不到的两件事
1. **确定性命理/历法计算**：排盘、真太阳时经度修正、起大运顺逆、流月流日定位、择吉日——这些是天文/历法精确运算。LLM 会**幻觉八字/大运，且每轮结果都不一样**。`dossier.seed.md` 作者已自认此事，用"手填正确八字绕过 LLM 自排盘的不可靠"。
2. **跨会话计数与命中率**：连续复盘天数、决策准确率、天机命中率——没有事件日志、没有计数器，模型只能**现编或回显手填文本**里的数字。

> 而文档**点名**预言验证（§4.8）与段位（§11）是"**让老板上瘾的核心机制**"。也就是说：**产品最值钱、最依赖可信数字的那一环，恰恰是当前最假的一环。** 聊天里说"你连续复盘 47 天了"——系统根本没在数。这不只是体验问题，对一个靠数字可信度建立信任的产品，是**信任崩塌**与潜在**虚假宣传**风险。

### 2.4 默认根本不发 prompt
默认 provider 是 `mock`（`server/src/llm/providers/mock.ts`，"零成本离线"），它**套用 DELIVERABLES 模板、完全忽略 system prompt**。所以默认/演示配置下，**整个 V6.0 prompt 层是惰性的**——任何天势行为都不会出现。V6.0 必须配**真实模型**（已接 `@anthropic-ai/sdk`）才谈得上。

**结论：这不是 prompt 调优能解决的，是一次"地基级"补齐——补一个确定性命理引擎 + 一个统一的持久状态层。14 个消费级功能都长在这两个地基之上；地基没好就先做上层，只会产出"显示假数字的卡片"，比不做更糟。**

## 四、优化 / 重构方向（依赖驱动，分 5 阶段）

**贯穿全局的设计铁律（一条规则解决 8 处重复问题）：**
> **一切数字、一切命理/历法运算，都必须在代码里算好、落库、再以结构化块注入 prompt；模型只负责"翻译成比喻 / 做战略映射 / 行文"。prompt 里明确禁止模型自算八字、自数天数、自报命中率。**

### Phase 0 — 决策与排雷（无代码，阻塞项）
1. **产品身份**：V6.0 单一军师是**替换**还是**并存**于 13-agent 智库/工坊？这决定 tab 结构、计费模型、strat 是"主角"还是"之一"。
2. **合规预检**：八字/预言/命中率在微信小程序属"算命/封建迷信"邻接内容，可能过不了平台审核与内容规定（团队已用"去米诺品牌 + 本命色委婉化"显露此担忧）。**这一项可能直接否决 P0 的命理引擎工作量**，必须先于工程拍板。
3. **建立 SSOT**：把 V6.0 prompt 纳入仓库版本管理（以 `strat.v6.md` 为源），终结"prompt 只活在生产 DB、无快照、无评审"的反模式（本文件即第一步）。

### Phase 1 — 地基（解锁一切，先于任何消费级功能）
- **A. 确定性命理引擎**：引入可信 ganzhi/真太阳时库（如 lunar-javascript / tyme4ts）+ 紫微（iztro）；新增 `BaziChart` 模型 + 采集页（阳历/阴历、时辰、性别、出生地——**当前 0 个此类输入**）；把推演结果作为结构化【命盘】块注入；**改写 prompt §4.2 从"内部完成推演"为"使用给定排盘结果，不要自算"**。
- **B. 统一持久状态层**：设计**一个** `StrategicArchive` 聚合（与现有 `GraphEntity/Memory` 协调，不要散成 8 张临时表）+ **会话结束自动写回**的 summarizer（替代复制粘贴 dossier）+ 统一的"算→存→注入"服务与单一注入块。
- **C. 调度器**：当前完全没有；任何基于时间的触发（久不复盘提醒、预言到期验证、里程碑/章节更新）都需要它。

### Phase 2 — 有状态核心（建在 Phase 1 之上，按此序）
1. **决策日志**（DecisionLog + 准确率计算）与**复盘**（Review/ReviewItem + 对齐率 + ReviewStreak）**先做**——因为预言命中率与段位都要消费它们的产物。
2. **预言验证**（Prophecy + 调度器驱动验证 + 命中率）。
3. **段位/里程碑/上瘾**（UserProgress：从 streak + 准确率 + 命中率派生 rank）。
> 全部**先把服务端数字做真**，UI 一律晚于数据；**数字没真之前，绝不在卡片/徽章上显示连续天数/命中率/段位**。

### Phase 3 — 编排与适配（依赖会话状态持久化）
- `Session.mode + routeStateJson` + 意图分类器（替代精确串匹配）实现 §3 路由。
- 十二问/六轮**状态机**（修复 >8 轮历史截断丢位）。
- 营收分阶段：用营收输入替换"融资轮次"问卷 + `deriveStage()` + 阶段配置驱动看板/轮次/八字深度。
- 角色语气由持久化的 `innerState` 驱动（与 `benmingColor` 解耦，本命色回归"纯品牌色"）。

### Phase 4 — 呈现与外化（最后做，只负责展示 1–3 的数据）
- **交付物分型**：给 Deliverable 契约加 `cardType/grade` + 渲染注册表（从 Notion 原页**还原被删的 12 种卡片骨架**）+ 前端按 cardType 分型组件。
- **数据看板**页（态势/KPI 趋势/五维雷达）。
- **命运叙事线/谶语**持久化 + 开局/授旗/年度复盘仪式页 + 可分享"天命速写卡"（裂变入口）。
- 前端**复盘/段位/命盘**页。
- 行业库结构化：补 §8 八字段，**美业与大健康拆分**（不同决策链/客单价），与 prompt 收敛到一处事实源。

### 横切 — 质量
- **禁用词 output-lint**（赋能/抓手/底层逻辑/颗粒度/范式转移）：在 `gateway.ts` 输出后挂一个非阻塞正则检查，命中写 `moderation_log(refType:'tone')`——**零依赖快赢**。
- 给 `evals` 播种 V6 专项用例：缺时辰、不信命理、禁用词、命盘一致性。
- **所有 V6 行为测试必须跑真实模型，绝不用 mock。**

## 五、需要先拍板的三件事（决定我下一步做什么）

1. **产品形态**：V6.0 单一军师 **替换** vs **并存** vs 暂不定、先做地基。
2. **命理合规**：命理/预言/命中率 **明着做** vs **弱化为"行业周期+个人经验"语言**（§16 已给降级话术）vs 需要先做平台/法务调研。
3. **计费模型**：日复盘的留存闭环与"每条消息计费"互斥——是否转向**订阅制**养一个常驻军师。

> 这三项任何一个变，Phase 1 的工作量和取舍都会变，所以应在动工前确定。
