# 全局审计报告 · V6.0 产品体系 × 当前系统

> **审计对象**：《天势终极版 V6.0》产品体系文档（Notion 导出稿）+ 当前代码库（app / server / admin）的落地现状。
> **审计口径**：只列问题、缺口、风险，不评优点。视角限定为**用户侧功能逻辑、交互流程、技术架构**三条线。
> **用途**：交由后续 agent 逐项处置。每条问题附【后果】和【证据】，可直接转化为任务。
> **审计日期**：2026-07-04

---

## 0. 总评（TL;DR）

**产品逻辑总评**：V6.0 是一份"prompt 即产品"的单体设计——把路由、状态机、账本、里程碑、计费全部押在一段 41KB 的提示词和用户的触发词上。它的核心价值（预言验证、决策准确率、段位、命运叙事线）全部依赖 30–365 天的留存数据积累，但文档对**冷启动期（前 30 天账本全空）没有任何设计**；它的核心卖点（命理精确到日的择时、称骨、谶语）**超出排盘引擎 v1 的实际能力**，且在微信小程序类目审核下有**合规生存风险**；它的商业承诺（"持续陪跑""外置大脑记忆完整"）与系统现实（token 计费轴、记忆按 agent 隔离、多 agent 无结论回流）存在**结构性矛盾**。

**技术架构总评**：服务端账本化（DecisionLog / ProphecyLog / ReviewLog / UserProgress / NatalChart / StrategicProfile）方向正确，但**账本只建了一半**：六层复盘只有 day 层落库、12 张 B 级卡只做 3 张、A 级报告模板未做、总军师→专业军师派单引擎未建、订阅消息推送未接、预言/决策的 LLM 抽取无人审核。最危险的一类问题是**"prompt 承诺了、服务端没接住"**——文档要求军师报"决策准确率、天机命中率、战略健康度、战斗力分"，其中一半有服务端权威数据，另一半（健康度、战斗力、五维评分）**没有任何数据来源定义，LLM 只能现编**，直接违反自家"服务端算数、AI 禁自算"铁律。

## 1. 产品逻辑问题（用户视角功能逻辑）

### P-1【严重】冷启动期产品空转：所有"上瘾机制"前 30 天无内容
- **问题**：预言验证（30 天后首验）、决策回顾（需已验证样本）、段位（14 天起步）、战友见证（需历史）、里程碑（第 7 天首解锁）——V6.0 的全部留存机制在新用户前 1–4 周**都是空账本**。文档第十一部分对此零设计。
- **用户后果**：新用户完成 6 轮诊断拿到报告后，第 2–29 天体验断崖：战报卡上"决策准确率 —%"、天机验证无预言可验，付费用户最容易流失的窗口恰好是产品最空的窗口。
- **证据**：文档 §11.2（首个里程碑 7 天）、§4.8（预言验证周期以月计）；服务端 `UserProgress` 注入规则"新用户零记录不注入"（AGENTS.md §10）——即军师在冷启动期连段位话术都无法引用。

### P-2【严重】预言验证的产品逻辑不可自洽，长期反噬信任
- **问题**：① 未命中话术"你的人谋胜了天时"（§4.8）是不可证伪的双标——命中归天机、不中归人谋，理性用户（企业主）识破后整个天势系统信用崩塌；② 元帅段位要求"预言命中率>50%"（§11.4），但预言由 LLM 生成、命中率不受产品控制，**段位体系最高层建立在随机变量上**；③ 月度 3–5 条预言的样本量，"本季命中率 67%"（n=3）无统计意义，把噪声当洞察反馈给用户。
- **用户后果**：用户要么被小样本"神准"误导做真实商业决策，要么在几次"双标解释"后弃用。
- **证据**：文档 §4.8、§11.4；服务端 `prophecyLog.ts` 命中率如实计算但无最小样本量保护。

### P-3【严重】命理内容的平台合规风险无降级预案
- **问题**：八字、紫微、称骨、择时、谶语是 V6.0 的第一卖点，但微信小程序对占卜/命理类内容有明确审核限制（封禁/下架先例大量存在）。文档的防呆机制（§16"老板不信命理"）只处理**用户**不信，完全没处理**平台**不允许：没有命理模块的开关化、没有"行业周期+个人节奏"话术的全局降级模式、没有审核期临时隐藏方案。
- **用户后果**：一次审核不通过即全量用户无法更新/使用；已付费用户权益承诺无法兑现。
- **证据**：文档 §4 全部、§16；前端 `packages/work/calendar`、`packages/work/gift` 命理 UI 无任何 feature flag。

### P-4【严重】"送你一卦"采集第三人生辰属敏感个人信息合规缺口
- **问题**：裂变机制要求老板提交**朋友的**出生年月日时、性别、出生地（§11.4），当事人未授权。生辰+出生地属于可识别的敏感个人信息，且产出的"天命速写卡"以链接公开可访问。
- **用户后果**：个保法合规风险；被算者投诉即成公关事件。
- **证据**：文档 §11.4；`packages/work/gift/index.tsx` 表单直接采集并出可分享链接，无当事人同意声明、无数据保留期限。

### P-5【高】交付物"晒朋友圈"与商业隐私直接冲突，无脱敏分享版
- **问题**：A 级报告和战报卡含营收数字、主要矛盾、命盘、决策准确率，话术却统一鼓励"截图发朋友圈""转给同行"（§交付物话术、§11.4）。没有"对外分享版"（隐去数字与命盘）与"自用完整版"的区分。
- **用户后果**：用户要么不敢分享（裂变失效），要么误分享泄露商业机密（信任事故）。
- **证据**：文档交付物规则总表"所有交付物底部…裂变引导"；`reportHtml.ts` 单一渲染版本，无脱敏变体。

### P-6【高】择时承诺"精确到日"超出排盘引擎实际能力，且无精度声明
- **问题**：文档 §4.5 承诺重大决策择时"精确到日"，§4.2 承诺称骨、从格判定。引擎 v1 现实：**无流日推演、称骨暂缓、从格/化格不处理、真太阳时无均时差、城市经度表仅 ~48 城、闰月前端无入口**。用户拿到的"吉日"实际是流月级近似，但产品话术以"精确"自居。
- **用户后果**：用户按"吉日"安排签约/开业，精度是假的；一旦用户对照其他排盘工具发现四柱都排错（时辰校正缺失），信任全失。
- **证据**：文档 §4.5/§4.2；AGENTS.md §13"排盘引擎 v1 已知边界"；`services/paipan.ts` 头注。

### P-7【高】阶段自适应依赖自报营收，无验证、无迁移机制
- **问题**：生存/起步/成长/扩张四阶段（§7）决定整个系统模式，但营收是用户口头自报，无任何交叉验证（经营三数回填与自报阶段不联动校验）；且文档没有定义**阶段迁移**——用户从生存期长到起步期，谁在什么时机把系统模式切过去？
- **用户后果**：阶段判断错一次，整套打法（复盘深度、看板配置、话术）持续错配，且没有纠正触发点。
- **证据**：文档 §7、§13"阶段适配"；`StrategicProfile.stage` 只有"认可方案+手动校准"两个回写点，无基于回填数据的阶段复核。

### P-8【高】行业知识库静态写死、覆盖窄、含真实公司名
- **问题**：仅 6 个行业模板 + 3 问兜底（§8），硬编码在 prompt 内无更新机制；"行业标杆案例"点名真实公司（榕树家、瑞斯国际），案例过时和名誉风险并存；行业客单价等数字写死在 2026 年的快照里会静默过期。
- **用户后果**：非 6 大行业的用户拿到的是通用话术+3 个反问，与"深度行业认知"的产品承诺不符。
- **证据**：文档 §8；admin 已有行业层（12 行业 survey）但与 prompt 内 6 行业模板**两套口径并存**，无同步机制。

### P-9【高】V6.0 文档与商业化体系完全脱节
- **问题**：文档通篇没有任何计费/权益设计：6 轮诊断是否耗额度、A 级报告是否收费、专业军师解锁与总军师动线的关系、免费用户能走到第几步——全部空白。现实是两轴计费（钻石解锁+token 月配额）+复盘保底 2 次/日，**产品文档与商业模型互不知道对方存在**。
- **用户后果**：额度用尽时"持续陪跑"承诺硬中断（保底只覆盖复盘前缀识别的调用，追问/悬念钩子兑现/月复盘超 2 次即被拦）；用户在第 5 轮诊断中途被 402 拦截是最坏的付费体验。
- **证据**：文档全篇无计费字样；`tokenQuota.ts` `REVIEW_GRACE_PER_DAY=2` 硬编码；memory《军师 strategy-system refactor》已标记"billing conflict→调度白名单"未解。

### P-10【中】六轮诊断无中途流失挽回与完成率设计
- **问题**：文档假设用户线性走完 6 轮（§6），没有：中途弃疗的召回策略、每轮的价值前置（用户第 2 轮就想要结论怎么办只有"3 轮快速通道"且触发靠军师自行判断"不耐烦"）、用户主动选择快速通道的入口。
- **用户后果**：诊断漏斗每轮流失无人管；快速通道的存在用户不可见。
- **证据**：文档 §6；scheduler 只有案卷闲置召回（48h），**无诊断中断召回**。

### P-11【中】"多业务线"承诺与单 active 案卷模型冲突
- **问题**：扩张期承诺"多业务线+博弈推演"（§7），但 Casefile 用户级唯一 active（V6.0 单主线叙事），多业务线老板无法并行追踪两条战线的军令与复盘。
- **证据**：文档 §7；AGENTS.md §10 "用户级 active 案卷唯一"。

### P-12【中】五维健康度/战略健康度/战斗力分无数据来源定义 → 伪量化
- **问题**：看板三层架构（§13）要求营收/客户/产品/团队/品牌五维 0–100 评分、战报卡要求"战斗力""战略健康度"百分数，但**没有定义任何一个分数的计算口径**。服务端只有 leads/consults/deals 三数与对齐率，其余分数只能 LLM 现编——违反系统自家"数字一律服务端算、AI 禁自算"的铁律。
- **用户后果**：用户拿到看似精确的假分数做经营判断。
- **证据**：文档 §13、B 级卡模板 `score-row`（战斗力/对齐率/完成率）；服务端仅 `alignRate` 有权威计算。

## 2. 交互流程问题

### F-1【严重】触发词路由作为主入口不可发现，且月/季/年复盘无 UI 入口
- **问题**：V6.0 的功能入口是聊天触发词（"这周"/"季度"/"帮我看看这个人"…，§3/§10）。前端虽已给日/周复盘做了结构化入口（执行页 tab），但**月复盘、季复盘、年复盘、团队复盘、决策择时、团队匹配、紧急决策**在 UI 上均无入口，用户必须"碰巧说对话"才能触发。产品能力的可发现性趋近于零。
- **用户后果**：付费买到的六层复盘体系，普通用户实际只会用到两层。
- **证据**：app 探索确认"月度复盘无独立 UI"；`resolveMode` 是确定性关键词匹配（见 A-2）。

### F-2【严重】悬念钩子无服务端兑现机制，会大规模失约
- **问题**：文档强制每次对话结尾埋钩子且"必须是真实的未完成分析"（§11.1），如"下周二来找我，我有个判断"。但服务端没有钩子登记表、没有到期提醒、没有下次对话自动带出——钩子兑现完全依赖 ① 用户记得回来 ② LLM 恰好记得自己埋过什么。
- **用户后果**：军师反复"说了不算"，仪式感设计反向变成失信设计，比不埋钩子更伤。
- **证据**：文档 §11.1；scheduler 三个 job（案卷闲置/复盘断档/预言到期）均无"钩子到期"类任务；无对应数据模型。

### F-3【高】里程碑解锁依赖用户当天到场，无补发机制
- **问题**：第 7/30/90/180/365 天解锁内容由军师在对话中给出（§11.2），但对话由用户发起。用户第 7 天没来，"一周天势微调"何时给、是否过期作废、第 35 天来了是先给 7 天的还是 30 天的——文档与服务端均未定义（`milestonesJson` 只记解锁日期，不管交付）。
- **证据**：文档 §11.2；`services/progress.ts` 里程碑按使用天数解锁但无"未交付内容"队列。

### F-4【高】提醒闭环整体缺失，且与微信订阅消息机制冲突
- **问题**：09:00 军令 / 21:30 复盘 / 周五周复盘三条日常提醒是陪跑动线的发动机，现状只接入了 21:30 复盘与报告生成两类微信订阅消息，09:00 军令与周五周复盘仍未落地。更深的问题是**微信订阅消息为一次性授权**——每收一条提醒消耗一次授权，高频日提醒需要用户每天重新授权，"每天 2 条提醒"在微信平台机制下**产品设计本身不成立**，需要重新设计触达策略（服务号模板消息/企微/短信分层），而不只是"接上推送"。
- **证据**：AGENTS.md §13 gap 2；`studio/index.tsx` REMINDERS 常量备注"订阅消息推送待接入"。

### F-5【高】六轮状态机脆弱：换会话即归零
- **问题**：诊断轮次 = 当前会话内 user 消息数（`services/context.ts`）。用户中途退出换了会话、误删会话、从战局页点了别的入口新开会话——轮次全部归零，军师重新开场采八字。多轮诊断的进度**没有用户级持久化**（StrategicProfile 存结论但不存"进行到第几轮"）。
- **用户后果**：最核心的 6 轮旅程可以被一次误操作清零，且军师毫无察觉地重新来过。
- **证据**：`context.ts:116-119` 轮次按 history 计算；`Session.mode` 存模式不存诊断进度。

### F-6【中】八字采集双轨分裂：主流程聊天采集，支线才有结构化表单
- **问题**：结构化八字表单只挂在天时日历/送一卦两个支线入口；**主诊断流程第 1 轮仍靠聊天文本报生辰**，LLM 解析"农历85年冬月初八早上"这类表述的出错率不可控，且与 NatalChart 结构化模型双轨并存。闰月后端支持、前端无入口，误差直接进命盘。
- **用户后果**：命盘排错 → 后续全部天势判断错，且用户无从发现。
- **证据**：app 探索报告 §2.8；AGENTS.md §13"阴历闰月前端采集 UI 暂未提供"。

### F-7【中】新用户首开 5 tab 中 3 个是空态，onboarding 漏斗未设计
- **问题**：战局页依赖军师判断（需诊断产出）、执行页依赖认可方案、智库页依赖资料上传——新用户首开只有对话 tab 有实质内容。文档只写了"新客户首次对话输出第 1 轮开场白"（§19），对"用户为什么会点进对话 tab、其余 tab 空态如何导流"没有设计。
- **证据**：`home/index.tsx`/`studio/index.tsx` 数据均依赖案卷与诊断产出。

### F-8【中】账本类数据"只进不出"：用户不可见、不可修正
- **问题**：决策日志、预言账本在前端无查看 UI（仅段位卡露出两个百分比）；军师会引用"你的快决策准确率 58%"，但用户**无法查看明细、无法对错记的决策提出修正**（AI 自动记账出错时没有申诉入口）。数据黑箱 + 不可纠错，与"镜子只呈现事实"的定位矛盾。
- **证据**：app 探索报告 §2.1/§2.2"无独立查看页/UI 未落位"；服务端有完整账本却无面向用户的读接口消费。

### F-9【中】报告分享链接无访问控制与生命周期
- **问题**：报告/卡片 HTML 凭不可猜 id 公开访问，链接一旦转出（聊天记录转发、截图带链接）即永久公开全部商业信息+命盘。无密码/过期/撤销机制，也无访问记录。
- **证据**：server 探索报告 §3.1"凭不可猜 id 公开访问"；`reportHtml.ts` 无 TTL/ACL。

### F-10【低】"战略档案让用户自己保存回传"的流程与系统现实矛盾
- **问题**：文档 §9.2 要求军师让用户"把这段档案发给我，我就能无缝接上"——这是无记忆单聊时代的设计。系统已有服务端记忆与档案注入，若 V6.0 prompt 原样入库，军师会向用户索要系统本来就有的东西，体验荒谬且暴露实现细节。
- **证据**：文档 §9.2 话术 vs `buildGenContext` 注入链（AGENTS.md ✦ 节）。

## 3. 技术架构问题

### A-1【严重】Prompt 与服务层职责边界未清理：LLM 仍可能伪造账本
- **问题**：41KB V6.0 全文作为总军师 systemPrompt 入库，其中包含大量**已被服务端接管的机制的操作指令**（预言日志格式、决策日志格式、段位规则、命中率计算、称骨）。LLM 按 prompt 指令有能力自行输出"预言日志 #7""你的命中率 67%"等伪账本数据，与服务端权威数据（注入的【天机账本】【决策账本】块）**同屏冲突**。Prompt 需要一次系统性的"去机制化"手术：只留角色/话术/方法论，删除所有数据生产指令。
- **证据**：`server/prompts/strat.v6.md`（41711 字节全文）；AGENTS.md §10 各账本"禁止 AI 自算口径"注入约束只覆盖了准确率，未覆盖账本条目本身。

### A-2【严重】模式路由是确定性关键词匹配，无意图分类兜底
- **问题**：`resolveMode` 靠触发词前缀/关键词（"这周""复盘""紧急"）判定六层复盘与紧急模式。中文口语表达高度多样（"昨儿的事咱盘盘""这个月咋样了"），漏匹配即走错模式：该落 ReviewLog 的复盘没落账本 → 连续天数断档 → 段位卡壳，**一次路由失误污染整条激励链路**。无 LLM 意图分类兜底、无落错后的补录机制。
- **证据**：server 探索报告 §5.5；`buildReviewPrompt` 确定性前缀识别同样用于计费保底判定（漏识别双重惩罚：不保底+不记账）。

### A-3【严重】记忆按 (用户×智能体) 隔离，与"外置大脑"承诺结构性冲突
- **问题**：Memory 表按 agent 隔离，总军师积累的用户认知，growth/intel/fund 等专业军师**一无所知**；且派单引擎（consult_specialist）未建、结论不回流。文档的单体假设（一个系统统揽）在多 agent 现实中不成立：用户在总军师处说过的关键事实，换个军师要重讲一遍——直接打脸"你比老板自己的记忆更完整"。
- **证据**：server 探索报告 §4.3"无显式 dispatch"；Memory 模型"按(用户×智能体)隔离"；AGENTS.md §13 gap 3/6。

### A-4【高】六层复盘只有 day 层有账本，月/季/年复盘将变成 LLM 现编
- **问题**：ReviewLog 仅 day 层由执行页落库，week/month/quarter/year"随 M3 触发词路由接入"（未做）。而月复盘流程要求核心数据审查、天机验证、决策回顾（§10）——没有 week/month 账本与 KPI 历史，这些环节的数字只能由 LLM 从对话上下文拼凑。
- **证据**：server 探索报告 §2.4；文档 §10 月复盘 5 步。

### A-5【高】交付物体系 2/3 缺失，第 6 轮"授旗仪式"无法兑现
- **问题**：12 张 B 级卡只上线 3 张（每日战报/天时日历/天命速写）；周/月/季战报、年度里程碑、紧急决策推演卡、晋升卡、性格操作手册卡、定位一页纸、十二问诊断卡、**A 级七章报告模板**全部未做。第 6 轮承诺输出 A×1+B×3，现在能给的只有 B×1。
- **证据**：AGENTS.md §13 gap 4；`services/cardHtml.ts` 仅 3 模板。

### A-6【高】预言/决策的自动记账质量不可控，无人审环节
- **问题**：预言靠 `gateway.extractProphecies` LLM 抽取异步收割，决策靠"认可方案分节标题启发式提取"。两条写入管道都没有：抽取质量抽检、用户确认环节、误抽的清理工具。账本是段位与信任体系的地基，地基数据是"LLM 抽 LLM"且无人审。
- **证据**：server 探索报告 §1.1/§1.2 写入源描述；AGENTS.md §10 PR-7"LLM 抽取随 PR-9 共建"（未见确认流）。

### A-7【高】scheduler 单实例 setInterval，扩容即重复提醒
- **问题**：进程内定时器，无分布式锁，幂等靠审计日志查重（脆弱且慢）。当前单实例部署成立，但任何水平扩容/双活切换都会双发提醒与召回。另外三个 job 都只"登记候选"，**候选→真实触达的下半场（订阅消息）整体缺位**，等于现在的 scheduler 只产日志不产用户价值。
- **证据**：`services/scheduler.ts`（setInterval 数组、审计去重）；server 探索报告 §2.3。

### A-8【中】计费保底与陪跑动线的缝隙未闭合
- **问题**：`REVIEW_GRACE_PER_DAY=2` 硬编码，覆盖面按"复盘前缀识别"判定。V6.0 动线中一天的正常交互（日复盘+追问 2 句+明日军令生成）就超 2 次；且悬念钩子兑现、里程碑解锁、月复盘这类"留存关键调用"都不在保底范围。额度用尽的用户会在最需要留住的动作上被 402。
- **证据**：`tokenQuota.ts:148`；P-9 关联。

### A-9【中】双行业口径、双 prompt 目录、品牌残留——内容资产管理混乱
- **问题**：① prompt 内 6 行业模板 vs admin 12 行业 survey 两套行业口径无同步；② `server/prompts/`（运行时）与 `server/src/data/prompts/`（基线存档）两目录并存待合并；③ V6.0 文档通篇旧品牌名违反工程品牌红线，入库前需逐段去品牌清洗（含 B 级卡模板里的品牌字样），目前只有 README 映射约定，无自动检查。
- **证据**：AGENTS.md §13 品牌残留条目 + §0#10；文档全篇。

### A-10【中】报告 HTML 依赖第三方字体 CDN（fonts.loli.net）
- **问题**：文档模板与自检清单强制 fonts.loli.net 字体（§A级自检清单末条）。第三方镜像可用性不受控；小程序 webview 的 downloadFile/request 域名白名单需要逐个配置，镜像换域名即全量报告字体退化。应内联/自托管字体子集。
- **证据**：文档 B 级卡骨架 `<link href="https://fonts.loli.net/...">`；自检清单"字体使用 fonts.loli.net"。

### A-11【中】十二问、目标阶梯、三势结论无结构化存储
- **问题**：`StrategicProfile.twelveQJson` 字段已预留但抽取管道未建（十二问答案散落对话）；目标阶梯（3-5年/年度/季度/本周）无模型，执行页为引导态；三势判断无 `forces` 结构化字段，战局页三势卡是静态框架。文档把这三者当核心资产反复引用（复盘对齐率、定位验证都依赖十二问结论），资产却没有落库路径。
- **证据**：AGENTS.md §13 gap 7/8；server 探索报告 §1.4。

### A-12【低】命盘 `chartJson` 全量快照 + 引擎版本复算的演进负担
- **问题**：排盘结果整块 JSON 存库、引擎升级需按 `PAIPAN_ENGINE_VERSION` 批量复算——但历史预言/择时建议是基于旧版命盘给出的，复算后**历史判断的依据变了**，预言验证的公平性没有定义（按当时命盘验，还是按新命盘验？）。
- **证据**：AGENTS.md §13"引擎升级须提版本并按版本复算"；ProphecyLog.basis 只存文本不存引擎版本。

## 4. V6.0 文档自身的质量问题（作为规格文档不合格）

### D-1 文档定位错位：它是一段提示词，不是产品规格
全文以第二人称对 LLM 说话（"你是……你要……"），路由表、状态机、计费边界、数据口径混在话术里。系统现实已经把账本、排盘、路由、计费移到服务端，**文档没有"prompt 负责什么 / 系统负责什么"的边界声明**，后续任何一方迭代都会与另一方漂移。需要拆分为：① 角色与话术规范（入 prompt）② 机制规格（入工程文档/SSOT）③ 交付物模板规范（入模板库）。

### D-2 内容错乱与导出事故
- B 级卡片⑨（天时日历）与⑫（性格操作手册）章节交错、内容互相穿插（原稿 881–955 行，两卡 HTML 片段互相嵌套）；
- 每日战报卡 HTML 中间插入"月度战报"标题行（725 行）；
- 列表编号多处接续错乱（第四部分性格手册从 6 开始编号、初始化行为从 4 开始、第八部分行业追问从 10 开始编号）；
- 交付物规则表与卡片清单为"（续）"，前半张表缺失。
直接按此稿移植模板会移植出错版卡片。

### D-3 关键机制只有话术没有规则
快速通道的触发条件（"老板不耐烦"如何判定）、悬念钩子的登记与兑现、里程碑的补发、段位降级（明确"只升不降"是服务端补的决定，文档未写）、天机命中率的最小样本、生存期用户能否使用择时/团队匹配——文档均以示例话术代替规则定义，无法直接工程化。

### D-4 品牌与红线冲突
通篇旧品牌标识，与仓库品牌红线冲突；文档还引用真实公司名做标杆案例（榕树家、瑞斯国际），存在名誉/侵权暴露。
